# ADR 002：將目標受眾擴展至混合團隊（開發 + 設計 + PM）

**日期：** 2026-06-20
**狀態：** 已接受

## 背景

最初的產品願景針對的是 5–15 人、沒有專屬 PM 的工程團隊。然而，實際使用模式顯示，即使是小型團隊也包含設計師和非技術背景的 PM，他們需要能掌握開發進度。這些使用者無法（也不應該需要）接觸如分支、PR 或 commit hash 等 git 概念。

我們考慮了兩個選項：
- **選項 A：** 保持純開發者導向，讓非開發者使用其他工具獲得可見性
- **選項 B：** 擴展至混合團隊，在相同的 git 原生核心之上增加簡化的「非開發者」視角

使用者選擇了**選項 B**。

## 決策

將產品願景擴展至服務三種角色——開發者、設計師和非技術背景的 PM——採用**雙 UI** 架構：

- **技術視角**（供開發者使用）：git 原生、感知分支/PR/commit、鍵盤驅動
- **簡單視角**（供非開發者使用）：以任務為基礎、無 git 概念、視覺優先

兩種視角共享相同的資料模型與狀態引擎。差異僅在於呈現方式與資訊密度。

### 已做出的變更

1. **使命：** 從「讓團隊把時間花在寫程式」擴大為「讓團隊把時間花在真正重要的工作上」——涵蓋非編碼角色
2. **目標使用者：** 在原有的 4 個開發者人物誌之外，新增 Sophia（設計師）和 Jamie（非技術 PM）人物誌
3. **團隊概況：** 更新為反映混合團隊（開發 + 設計 + PM）
4. **核心原則：** 新增第 6 條原則——「Dual UI：同一份數據，兩種視界」
5. **當前重點：** 將 Dual-View 架構列為 P1 項目，Figma 整合和 Kanban 視圖列為 Next 項目

## 影響

**正面：**
- 大幅擴大可觸及市場（TAM 涵蓋非開發者角色）
- 設計師和 PM 成為工具在團隊內的推廣者
- Dual UI 成為相較純開發工具的一大競爭優勢
- 「簡單視角」為新團隊成員提供更順暢的上手路徑

**風險（已緩解）：**
- 範圍蔓延風險：我們透過將 Dual UI 保持為純呈現層來避免此風險——而非具有不同功能的第二個產品
- 稀釋開發者優先身份的風險：透過將開發者體驗作為首要任務來緩解（開發者仍是主要使用者，透過 git 驅動任務狀態）
- 過度投入 Figma 整合的風險：將其推至「Next」而非「Now」——我們先交付核心功能再進行整合

## 備註

- Figma 整合和 Kanban 視圖明確歸類為「Next」項目，而非 P1——核心 git 自動化始終為 P0
- Dual UI 原則必須一致地應用：相同的資料，不同的表面。不應為不同角色建立各自的 schema 或工作流程。
- 未來的 plugin/UI 擴充點應在設計中考量兩種視角
